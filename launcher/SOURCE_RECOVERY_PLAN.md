# Восстановление лаунчера без исходников

Ситуация: исходники лаунчера потеряны, есть только собранный APK из архива `Под 64.zip`.

## Главный вывод

Полностью восстановить исходный проект Android Studio в исходном виде из APK нельзя. После сборки теряются:

- нормальные имена части классов и методов, если была обфускация;
- комментарии;
- структура Gradle-проекта;
- часть исходной архитектуры;
- история коммитов;
- удобная структура модулей.

Но можно восстановить рабочую базу для продолжения разработки:

1. вытащить ресурсы и layout из APK;
2. восстановить AndroidManifest и список activity/service/provider;
3. восстановить визуальную структуру экранов лаунчера;
4. создать новый чистый Android-проект;
5. заново написать нормальную логику лаунчера;
6. оставить старый APK как технический референс.

## Что уже известно по текущему APK

- Package: `com.break.russia`
- Version: `16.26.7.0`
- minSdk: 24
- targetSdk: 34
- Launcher activity: `com.blackhub.bronline.launcher.activities.MainActivity`
- Game activity: `com.blackhub.bronline.game.core.JNIActivity`
- ABI: `arm64-v8a`, `armeabi-v7a`
- В APK есть Firebase, Crashlytics, AppMetrica, Google Billing, RuStore Billing и WorkManager.

## Что можно восстановить из APK

### Можно восстановить хорошо

- иконки приложения;
- изображения и drawable-ресурсы;
- layout-файлы;
- названия экранов и основных компонентов;
- AndroidManifest;
- список разрешений;
- native-библиотеки;
- часть строковых ресурсов;
- примерный UI-поток.

### Можно восстановить частично

- Java/Kotlin-код через decompile;
- модели данных;
- навигацию между экранами;
- сетевые вызовы;
- логику обновления клиента;
- обработку ошибок.

### Нельзя восстановить нормально

- исходный Gradle-проект один-в-один;
- комментарии разработчика;
- чистую архитектуру;
- приватные ключи подписи;
- историю разработки;
- нормальные имена после обфускации;
- секреты и доступы, если они не были сохранены отдельно.

## Важный момент про подпись APK

Если потерян keystore/ключ подписи, обновлять старое приложение поверх установленного APK будет нельзя. Android считает APK с другим ключом другим издателем.

Варианты:

1. найти старый keystore;
2. если приложение публиковалось через Google Play/AppGallery/RuStore — проверить возможность восстановления/смены ключа через площадку;
3. если ключ потерян окончательно — выпускать приложение как новый пакет или делать миграцию через сервер/инструкцию для игроков.

## План восстановления

### Этап 1. Инвентаризация APK

- сохранить исходный ZIP/APK и SHA-256;
- разобрать AndroidManifest;
- составить список activity/service/provider/receiver;
- составить список layout, drawable, assets, lib;
- определить, какие ресурсы относятся именно к лаунчеру, а какие к игре.

### Этап 2. Новый чистый проект

Создать новый Android-проект:

- Kotlin или Java;
- Gradle;
- minSdk 24;
- targetSdk 34+;
- package на выбор: оставить `com.break.russia`, если есть право и ключ подписи, или выбрать новый package;
- architecture: MVVM/MVI;
- modules: `app`, `launcher-core`, `network`, `updater`.

### Этап 3. UI лаунчера

Сделать новый главный экран:

- `Играть`;
- статус сервера;
- новости;
- прогресс обновления;
- кнопка `Проверить файлы`;
- кнопка `Настройки`;
- понятные ошибки.

### Этап 4. Обновления клиента

Реализовать нормальный updater:

- manifest файлов на сервере;
- versionCode/versionName;
- размер файла;
- SHA-256;
- докачка;
- повтор после ошибки;
- запрет запуска при повреждённых файлах.

### Этап 5. Запуск игры

- проверить наличие нужных файлов;
- проверить ABI устройства;
- подготовить environment;
- запускать игровую activity только после успешной проверки;
- логировать ошибку запуска.

### Этап 6. Безопасность

- не хранить секреты в APK;
- убрать лишний HTTP;
- использовать HTTPS;
- минимизировать разрешения;
- использовать app-specific storage;
- не делать обход оплаты, лицензий, античита или защиты сервера.

### Этап 7. Релиз

- собрать debug APK;
- протестировать Android 7–14;
- собрать release APK/AAB;
- подписать новым или восстановленным ключом;
- подготовить инструкцию для игроков.

## Что делать сейчас

Самый правильный следующий шаг: создать новый Android-проект в репозитории и начать с MVP-лаунчера, а старый APK использовать только как источник ресурсов и технического сравнения.

MVP должен уметь:

1. открываться;
2. показывать главный экран;
3. показывать статус сервера;
4. показывать новости;
5. проверять файлы;
6. запускать игру только после успешной проверки.
